## Architecture Decisions

- The canvas syncs through the official `@liveblocks/react-flow` integration (`useLiveblocksFlow`) rather than hand-rolling React Flow ↔ Liveblocks Storage with `useStorage`/`useMutation`. The hook owns the CRDT mapping (one `LiveMap`-backed diagram under storage key `"flow"`) and returns React Flow-ready `nodes`/`edges`/`onNodesChange`/`onEdgesChange`/`onConnect`/`onDelete`. We run it in `suspense: true` mode under `ClientSideSuspense`, so `nodes`/`edges` are always ready arrays inside `FlowCanvas` (no `isLoading` branch). `CanvasNodeData` needs an index signature because xyflow's `Node<T>` constrains `T extends Record<string, unknown>`. Connection-time errors surface through a small dependency-free class `CanvasErrorBoundary` (react-error-boundary isn't a dep) rather than crashing the editor.
- Liveblocks authenticates with **access tokens** (`prepareSession` + `userInfo`), not ID tokens, because the feature spec requires the session token itself to carry the user's name/avatar/cursor-color — that metadata rides on the access token, whereas ID tokens carry only identity and resolve user info client-side. The room id is the project id, so authorization reuses `findAccessibleProject` (the same owner-or-collaborator check as the project/collaborator routes) and returns **403** on denial; the room is created with `getOrCreateRoom(..., { defaultAccesses: [] })` so it's private and reachable only through issued session tokens. No `publicApiKey` is used — the browser hits `/api/liveblocks-auth`, which holds the secret key server-side.
- The Liveblocks node client is created **lazily** (`getLiveblocks()`), unlike the eager `lib/prisma.ts`. Prisma can be eager because `DATABASE_URL` is always set; the Liveblocks secret may be absent until the user provisions a dashboard project, and `next build` evaluates API-route module scope — an eager throw broke the build. Lazy creation defers the secret requirement to first token issuance while still caching one client on `globalThis`.

- Collaborator mutations enforce ownership the same way project renames/deletes do — through an owner-scoped atomic Prisma write (`project.update({ where: { id, ownerId }, data: { collaborators: { create | deleteMany } } })`) rather than a separate read-then-check. A non-owner's write matches no row → P2025 → 404. Listing collaborators is allowed for any member (owner or invited), so the GET handler uses `findAccessibleProject` and returns `isOwner` so the client knows whether to render management controls; the boundary is still the server, not the flag. Collaborator avatars/names are enriched from Clerk's Backend API at read time (no local user table per the spec); enrichment degrades to email-only on a per-email miss or a whole-batch API failure.
- The share dialog loads its list via an explicit `reload` triggered from the dialog's `onOpenChange` (open) instead of a `useEffect` keyed on an `enabled` prop — this avoids the `react-hooks/set-state-in-effect` lint rule (the async loader sets `isLoading` synchronously) and matches the project's event-driven mutation hooks (`use-project-actions`). The Share trigger + dialog are one self-contained client component rendered in the navbar, mirroring how the project dialogs co-locate trigger and dialog.
- Workspace chrome spans the layout/page boundary: the navbar + project sidebar live in the shared `/editor` layout (`EditorShell`), but workspace-specific bits (project name, current-room highlight, Share/AI-toggle, AI panel) belong to a single project. Rather than parallel routes, `EditorShell` derives the active project from `useParams().projectId` against the lists it already fetched, and an `AiSidebarProvider` context bridges the navbar's AI toggle (shared layout) to the AI panel (workspace page). The page stays a server component for the access gate; the AI panel is the only client piece it renders.
- Access resolution is one composed helper (`lib/project-access.ts` `resolveProjectAccess`) returning a discriminated `unauthenticated | denied | granted` union, so the server page switches on it without re-deriving auth. The identity (`getCurrentUserContext`) and access-check (`findAccessibleProject`) primitives stay in `lib/auth.ts` / `lib/projects.ts`; `project-access.ts` only orchestrates. Missing and unauthorized both map to `denied` → `AccessDenied` (no existence leak), matching the API's 404 collapse.
- Prisma 7 keeps the connection URL out of `schema.prisma` (datasource has provider only); `DATABASE_URL` is wired through `prisma.config.ts` for the CLI and resolved at runtime in `lib/prisma.ts`, which branches on the URL scheme — Accelerate (`prisma+postgres://`) uses `accelerateUrl` + `withAccelerate()`, everything else uses the `PrismaPg` direct adapter (an Accelerate URL must never be handed to a driver adapter). The generated client is imported from `@/generated/prisma/client.js` (not `@prisma/client`); the dir is gitignored + regenerated on `postinstall`.
- Schema is multi-file: `prisma.config.ts` `schema` points at the `prisma` directory, so `prisma/schema.prisma` (generator + datasource) and `prisma/models/*.prisma` (models) are merged. Note: `prisma migrate dev` did not pick up the regenerated client reliably after the schema swap — run `prisma generate` explicitly if the client looks stale (`P2022` on a dropped column).
- Collaborators are modeled by **email** (`ProjectCollaborator.email`), not Clerk user ID — invites can precede the invitee having an account. `Spec`/`TaskRun` tables from the architecture storage model are deferred to a later phase (not in the 05-prisma spec).
- Project dialog state is owned by one `useProjectDialogs` hook instance, lifted into `ProjectDialogsProvider` (rendered inside `EditorShell`) and shared via context. Both the sidebar and the editor home screen open the same dialogs, which are rendered once at the top of the editor tree rather than duplicated per trigger.
- Editor chrome lives at `/editor`, not the root route. The root layout holds only `ClerkProvider` + fonts; the `EditorShell` (navbar + sidebar) is the layout for the `/editor` segment so auth pages and the `/` redirect render without editor chrome.
- Clerk URL env vars added to `.env.local`: `NEXT_PUBLIC_CLERK_SIGN_IN_URL`, `NEXT_PUBLIC_CLERK_SIGN_UP_URL`, and the two `*_FALLBACK_REDIRECT_URL` vars (both `/editor`). Standard Clerk names — none invented.
- `@clerk/ui` v1.15 appearance schema uses `colorForeground`/`colorPrimaryForeground`/`colorInput`/`colorMuted` naming (not the older `colorText`/`colorInputBackground`).
